# Dawn TOD 体积云模块美术方向评审

> 评审日期：2026-07-21  
> 评审范围：当前 URP 体积云 Shader、Volume 参数面板、默认资源，以及 `VolumeCloud` 测试场景的实时画面。  
> 本文只定义美术表现问题、调参体验问题与优化优先级，不讨论具体技术实现方案。

## 1. 总体结论

当前模块已经具备“体积云原型”的完整轮廓：能够生成三维云体，有基础自阴影、主光方向响应、远近遮挡和动态噪声，也已经开始把 Shape Size、Detail Frequency、Density Bias 转换成相对友好的美术语义。

但从商业项目的最终画面和美术生产效率来看，目前更接近“技术原型后期 / 美术生产工具前期”，还不能作为稳定的成片级体积云方案。主要差距不在于云是否存在，而在于：

1. 光照能量不足，白云容易整体发灰，厚云内部尤其灰闷。
2. 云体以大块、扁平、连片的形态为主，缺少清晰的云型语言和从大形到碎边的层级。
3. 参数面板仍暴露大量底层变量，美术无法用明确目标快速调出“晴天积云、阴天层云、暴风云”等结果。
4. 云与天空、太阳、地面和天气系统之间的视觉联系不足，容易像一层独立叠加的灰色体积。
5. 当前低分辨率表现偏软，运动中的稳定性与远景质量仍需要专项验收。

当前最值得优先解决的不是增加更多参数，而是先建立三件事：**干净而有层次的白云光照、可辨认的云型、以美术目标组织的控制面板。**

## 2. 关于“颜色全白但云仍然灰蒙”的判断

### 结论

是的，**缺少多重散射或等价的内部光能回填，是当前白云发灰的重要原因之一**；但问题不只来自多重散射。

当前两个云色都设为白色时，它们表达的是参与光照计算的颜色倾向，并不等于最终画面直接输出白色。云色仍会继续受到以下因素压暗：

- 光线穿过云体时的吸收与自阴影；
- 观察方向与太阳方向形成的相位响应；
- 云体本身的密度和厚度；
- 主光强度与颜色；
- 最终曝光与色调映射。

当前画面主要依靠太阳直射光照亮云体，缺少足够的天空环境光、地面反弹光和云内多次散射带来的亮度回填。结果是薄边还能偏亮，厚云和背光面很快掉入灰色，但又没有足够丰富的冷暖与明暗层次，于是呈现为“均匀灰蒙”，而不是有重量感的明暗。

因此，单纯把 Color A / Color B 调成白色，只能保证不主动染色，不能解决云体的亮度和通透感。继续单独提高颜色亮度也只能暂时抬灰，容易同时损失阴影层次和日出日落的染色能力。

### 当前画面中可见的具体表现

- 白云高光不够干净，主体亮度与阴影亮度距离较近。
- 厚云底部是大面积软灰，缺少天空补光形成的通透层次。
- 朝向太阳的一侧没有形成足够明确的亮边或银边识别。
- 背光面、侧光面和云底之间的光色关系不明显，体积感更多来自密度轮廓，而不是光照塑形。
- 当前相位响应的控制方式和参数组合不够直观，容易让顺光、侧光、逆光的亮度变化不可预测。

## 3. Shader 最终美术表现问题

### 3.1 云体大形偏“片”和“块”

当前实时画面中，大云体容易形成横向连续的顶棚或厚片，小云体则偏圆钝团块。它们能被识别为云，但还缺少商业画面常见的结构语言：底部相对平稳、主体有向上生长趋势、迎光面饱满、背光面有重量、边缘逐级破碎。

现有大形、中形、细节之间的尺度差不够清楚。测试配置中的大形尺度很大，而所谓“细节”反而比基础形状更低频，这会进一步放大块状和模糊感。最终结果是轮廓有变化，但近看缺少可信的内部组织，远看又容易连成一整层。

### 3.2 缺少稳定、可辨认的云型

当前主要表现为同一套噪声形态的疏密变化，尚未形成明显不同的云型族群。例如晴天积云、厚重层云、降雨云、塔状对流云，不能仅靠当前面板快速得到稳定且可重复的区别。

这会直接影响天气叙事：晴天只像“云少一些”，阴天只像“云多一些”，暴风天气只像“更黑更密”，而不是在轮廓、厚度、垂直发展和光照性格上产生整体变化。

### 3.3 云层空间感和远景关系不足

当前云被限制在固定的盒状区域中，画面上容易出现统一高度、统一厚度和大面积平顶/平底的感觉。靠近地平线时，云层缺少足够自然的曲率、距离衰减和大气包裹，容易出现“悬在场景上方的一层体积片”。

区域边界虽然做了淡出，但对商业镜头来说仍需重点检查：长焦镜头、低机位、飞行镜头和大范围移动时是否能看到边界、重复图案或突然消失。

### 3.4 光照缺少层次，而不仅是亮度不足

当前最显眼的问题是暗，但更深层的问题是光照语言不完整：

- 直射光、天空光、云内散射和云底反照没有形成清楚分工；
- 顺光、侧光、逆光缺少各自鲜明而稳定的美术性格；
- 高光、亮部、中灰、暗部之间缺少可控的层级；
- 日出日落时容易整体一起染色，难以保留亮边暖、阴影冷、云底沉的综合色彩关系；
- 厚云的暗部容易变成无细节的灰团，而不是有深度的阴影。

### 3.5 轮廓偏软，局部细节层级不足

当前默认采用较低分辨率渲染，实时画面中的轮廓和云内阴影都有明显的柔化倾向。柔和本身符合云的材质特性，但目前容易从“自然柔软”滑向“分辨率不足的糊”。

尤其需要关注：

- 小云块是否被放大成棉花团；
- 近景碎边是否缺少细小侵蚀；
- 云与几何体交界处是否出现软边、漏色或光晕；
- 镜头运动时是否出现噪点固定在屏幕上、闪烁、拖影或边缘跳动。

### 3.6 动态容易表现为平移或沸腾

现有动画能够让云动起来，但美术上尚缺少清晰的风向、整体输运、缓慢生消和局部演化之间的层次。不同噪声层速度差距较大时，细节容易从云体表面滑过；对三维噪声轴向的持续偏移，也可能让云体出现不自然的垂直沸腾。

商业镜头应避免两种典型观感：整张云图横向贴图式滑动，以及轮廓原地快速翻滚但云团没有真实位移。

### 3.7 与场景和天气的融合不足

当前云主要完成了“云叠加到天空上”，但尚未完整承担天气系统中的环境角色：

- 地面缺少与云量和云密度对应的软阴影变化；
- 太阳被厚云遮挡时，环境亮度和光色缺少同步响应；
- 云层与大气透视、雾、地平线颜色之间的关系不够紧密；
- 雨云还没有明确的降水区、深色云底和远处雨幕联系；
- 夜间和月光条件下是否仍能保持可读层次，当前没有足够证据。

## 4. 参数面板问题

### 4.1 面板以底层变量为中心，不是以美术目标为中心

当前面板中的 Shape Noise Weights、Detail Weights、Detail Noise Weight、Color Offset 1/2、Phase Parameters、Speed Warp，以及 Ray Step Exponent / Length 等参数，需要使用者先理解 Shader 内部关系，才能猜测画面结果。

这些参数允许技术美术精调，但不适合作为一线美术的主要工作界面。尤其是多个值打包在 Vector4 中时，单个分量没有独立含义、范围和风险提示，调参结果不可预测，也很难在团队中口头交流和复现。

### 4.2 缺少最核心的直接控制

当前没有足够直接、独立的美术控制来回答以下常见需求：

- 我想要多少云，而不是整体加多少密度；
- 我想要哪一种云，而不是换一组噪声权重；
- 我想让云更高、更厚、更有垂直发展；
- 我想保留大形，只增加或减少碎边；
- 我想让白云更亮，但不洗掉阴影；
- 我想增加天空补光、压暗云底或加强逆光银边；
- 我想明确设置风向、整体速度和形变速度；
- 我想从晴天平滑过渡到阴天、暴雨，而不重新调一整套底层参数。

其中“云量/覆盖率”和“云型”尤为关键。当前 Weather Map 虽然命名上包含覆盖含义，但实际美术反馈更接近高度形态控制，不能替代直观、稳定的覆盖率控制。

### 4.3 参数相互耦合，容易调崩

Density Bias、Density Multiplier、Shape Noise Weights、Detail Noise Weight 和 Height Weights 会共同决定云是否存在、是否连片、是否被侵蚀。美术调整其中一个值时，常常同时改变覆盖率、轮廓、厚度和明暗，难以只修一个问题。

Color A / B 与两个 Color Offset 的角色也不清楚。使用者很难从命名判断哪个控制亮部、阴影、云底或太阳染色，更无法理解为什么颜色设为白色仍会发灰。

质量参数同样存在耦合：当前是分辨率、步数、步长和指数的组合，而不是“低、中、高、电影级”等可预期的视觉档位。美术很难判断性能变化会牺牲轮廓、阴影还是远景距离。

### 4.4 缺少预设、预览和诊断反馈

当前面板缺少能显著提升美术生产效率的辅助能力：

- 代表性云型/天气预设；
- 参数恢复和安全范围提示；
- 对当前云量、密度、侵蚀和光照结果的可视化预览；
- 对异常组合的提示，例如细节尺度比大形还大、相位响应超出常用范围、采样不足或云层边界可能进入镜头；
- 同一参数在晴天、日落、夜间和暴风天气下的快速对照。

### 4.5 面板层级需要重新划分

当前所有参数基本处于同一编辑深度。商业美术界面至少需要区分：

- 日常必调：云量、云型、位置与厚度、密度、亮度、阴影、风；
- 造型精调：大形、中形、碎边、垂直生长、柔软度；
- 光照精调：环境补光、太阳响应、银边、云底、色彩关系；
- 高级与调试：纹理资源、底层权重、相位细节和采样参数。

本阶段的问题不在于具体采用哪种 UI，而是必须避免让普通美术直接面对原始 Shader 参数作为唯一入口。

## 5. 当前资源与默认配置问题

1. 当前测试配置的大形尺寸过大，细节频率又低于基础形状，直接造成大块、扁平和软糊的观感。
2. 默认细节纹理命名和内容来源不够像经过美术验证的云专用资源，容易产生不可控的重复或特殊形状残留。
3. Weather Map、Mask 和其他测试贴图混放在运行时默认资源目录中，包含明显的实验资源；这会削弱默认效果的可信度，也容易让使用者误选。
4. 天气图目前主要使用单通道信息，无法从面板上理解各通道是否代表覆盖、云型、高度或降水。
5. 默认参数更像迁移自某个源场景的特定数值，不像经过多天气、多光照和多镜头验证的产品默认值。

商业模块的默认状态必须首先“开箱好看且不容易坏”。高级用户可以继续深入，但普通用户不应从一组实验性资源和魔法数字开始。

## 6. 美术优化优先级

### P0：成片质量的基础门槛

1. 解决白云整体灰蒙，建立直射光、环境补光、云内亮度与阴影之间的清晰层次。
2. 建立可辨认的大形—中形—碎边层级，消除大面积片状、顶棚状和棉花团观感。
3. 提供稳定的云量与云型表达，让晴天、阴天、暴风天气不再只是密度不同。
4. 重整主要参数语义，让美术能围绕画面目标工作，而不是围绕 Shader 变量试错。
5. 重新确定一套可作为产品门面的默认资源和默认参数。

### P1：商业场景融合与稳定性

1. 加强云与天空、大气、太阳、雾和地面的整体光色联系。
2. 补足云对地面和环境光照的影响，使天气变化能影响整个场景。
3. 验证并改善近景、远景、地平线、低机位和飞行镜头中的空间可信度。
4. 改善低分辨率下的轮廓、几何交界和运动稳定性。
5. 建立自然的风向、整体移动和缓慢形变层次，降低滑动与沸腾感。
6. 提供一组可直接使用且可平滑过渡的代表性天气预设。

### P2：风格化与制作效率

1. 扩展日出、日落、夜间、月光、风暴等风格化光照范围。
2. 增加适合动画、开放世界、展示镜头等不同用途的美术模板。
3. 增加调试视图、对比预览和质量档位，缩短美术迭代时间。
4. 为特殊需求保留高级底层控制，但默认折叠，不干扰日常制作。

## 7. 建议的美术验收标准

后续版本是否达到商业可用，不应只看参数数量或单张截图，建议至少满足以下画面标准：

1. 正午白云在正常曝光下有干净亮部、有层次中间调和可读暗部，不整体灰蒙，也不过曝成白片。
2. 同一云层在顺光、侧光、逆光下具有明确且自然的差异，逆光可形成受控银边。
3. 晴天积云、层云、厚重雨云至少能在轮廓、厚度、垂直发展和光照性格上被一眼区分。
4. 地平线、低机位和大范围移动中看不到盒状边界、明显重复或整层平板感。
5. 连续观察 30 秒，云团有一致风向和整体位移，细节不会明显贴图滑动或原地沸腾。
6. 云量变化能够同步影响太阳遮挡、地面阴影、环境亮度和天气氛围。
7. 低、中、高质量档位保持同一美术风格，只改变可接受的细节与稳定性，不改变云型和明暗性格。
8. 普通美术能够在不理解相位函数、噪声通道和步进参数的前提下，快速得到稳定结果。

## 8. 本轮评审的最终判断

当前模块的基础路径是成立的，已经能生成可用的体积轮廓，也有继续打磨的价值。现在最影响商业感的并不是“少一个特效”，而是**光照能量、云型结构和美术控制语义三者尚未闭环**。

对白云发灰问题的判断可以明确为：**多重散射不足是重要原因，但还叠加了环境补光缺失、厚云自阴影过强、相位响应不够稳定，以及缺少独立亮度层级控制。** 如果只补一个颜色参数，画面仍很难从灰色体积提升到商业级白云；必须以完整的光照层次作为一个整体问题处理。
